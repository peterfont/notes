# Agent Loop 常见反模式与三道保险

## 核心观点

Loop 不是把错误结构重复十遍。可靠的 Loop 必须让每一轮都产生新的证据，并且主动验证真实结果。

## 完整整理稿

你让 AI 发布一篇文章，它忙了十轮，最后说成功；但你打开后台，草稿箱还是空的，积分却少了一大截。问题不一定是模型不够聪明，而是 Loop 把错误结构重复了十遍。

### 1. 状态污染

第一轮上传封面失败，AI 却把“封面已上传”写进任务状态。下一轮它不再检查后台，直接创建草稿；第三轮又基于不存在的草稿生成链接。一次工具错误只影响一步，但错误状态会污染后面每一步。

修复方法是：关键事实只能从外部系统重新读取，并附上截图、文件路径或后台 ID 作为证据。

### 2. Token Burn：无效空转

AI 反复读取同一篇文章、重写相同计划、再次调用已经失败的上传接口，却没有增加任何新证据。

判断是否要继续，只问一个问题：这一轮产生 Evidence Delta（证据增量）了吗？有没有新的错误码、新的后台状态，或者新的产物？如果没有新证据，就应该停止空转，先改变方法。

### 3. Soft Verification：软验证

执行者说“应该已经发好”，验收者看完这句话就回复“检查通过”。这不是验证，只是两个 AI 互相相信。

Hard Verification（硬验证）要求绕开描述，直接打开后台，逐项检查标题是否存在、正文能否打开、封面是否显示。验证者的职责不是证明执行者没错，而是主动寻找能够推翻“成功”的证据。

### 4. 假成功

上传接口返回成功、封面文件已经生成、后台页面也能打开，这些都只说明中间步骤成功，不代表文章真的进入草稿箱。

最终必须回到 Acceptance Criteria：标题正确、正文完整、封面可见、草稿 ID 可以再次查询。工具成功只是过程信号，用户要的结果才是最终闸门。

## 给 Loop 加三道保险

1. **外部状态：** 把任务、产物和错误写入文件或后台，不依赖聊天记忆。
2. **证据增量：** 每一轮必须带回一个能改变判断的新事实。
3. **按验收条件检查：** 连续失败三次就触发 Circuit Breaker（断路器），停止继续消耗 Token，报告卡点并转人工定位。

Loop 的价值不是让 AI 一直做，而是让每一轮都更接近证据。
