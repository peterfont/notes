# Loop Engineering：让 AI 对结果负责

> 整合来源：AI酋长 Andy 的 6 篇口播文案（原始稿保留在 `00_收集箱/AI酋长Andy/`）

## 一、为什么 Prompt 不够

Prompt 适合让 AI 在当前步骤生成答案，但现实任务往往包含多个步骤、外部工具和不断变化的状态。发布文章时，AI 可能已经写完正文，却因为登录过期、封面上传失败或接口变化，根本没有创建出真实草稿。

因此，Prompt 让 AI 说出答案，Loop 让 AI 对结果负责。Loop 会持续观察状态、规划下一步、调用工具、读取结果并修正路线，直到验收条件被真实满足。

## 二、一个可靠 Loop 的四个阶段

### Discover：发现状态

先确认账号是否登录、素材是否齐全、后台是否已有草稿。不要从假设开始。

### Plan：规划动作

明确下一步做什么、顺序是什么、需要什么输入，以及完成后用什么证据检查。

### Execute：执行动作

真正上传文件、写入内容、调用接口、创建任务或草稿。

### Verify：验证结果

重新打开真实系统，检查结果是否存在。验证失败，就把错误带回下一轮，而不是继续沿用旧状态。

## 三、Loop 必须有验收和停止

**验收条件（Acceptance Criteria）** 是“什么才算完成”：标题正确、正文完整、封面可见、草稿 ID 可以再次查询。

**停止条件（Stop Condition）** 是“什么时候必须停”：连续失败三次、遇到规则冲突、数据缺失或超过预算，就暂停并报告卡点。

没有验收条件，AI 容易假成功；没有停止条件，AI 容易无限重试。

## 四、四种常见反模式

1. **状态污染：** 把工具失败写成成功，后续步骤继续基于错误状态执行。关键事实必须从外部系统重新读取，并留下 ID、路径或截图。
2. **Token Burn：** 重复读取、重复规划、重复调用失败接口，却没有产生新证据。每轮都要有 Evidence Delta（证据增量）。
3. **软验证：** 执行者说“应该成功”，验收者就相信。硬验证必须直接打开后台，逐项检查结果。
4. **假成功：** 中间接口返回成功，不等于用户要的结果已经完成。最终必须回到验收条件。

## 五、验证决定交付质量

批量生成商品页、任务或内容时，错误会复制扩散。验证不能只放在最后，而要放在每个关键动作之后，用更早验证、更小批次的方式降低返工成本。

验证至少要留下三层证据：

- **结果证据：** 价格、标题、图片、正文是否正确；
- **过程证据：** 来源、权限、禁用词和执行路径是否合规；
- **状态证据：** 后台是否真的发布，之后还能否用真实 ID 查询。

验证器要主动寻找推翻成功的证据，并为每个关键项输出 Pass 或 Fail。连续失败三次就触发 Circuit Breaker，停止继续烧 Token，转人工定位。

## 六、什么任务值得升级成 Loop

可以看四个信号：是否需要调用工具，任务状态是否会变化，过程中是否存在不确定性，最终结果是否能硬验证。

例如把周会录音变成真实任务：AI 先提取候选行动项，再补齐负责人、截止时间和验收标准；人工确认高风险内容后，系统才把任务写入外部任务系统，并返回真实任务 ID。

生成会议纪要只是文字结果，创建了负责人明确、时间明确、状态可追踪的真实任务，才是工作结果。

## 七、AI 自主行动的边界

自动化不应按“整个任务”一刀切，而应按每个动作的风险分级：

- **绿区：** 低损失、可撤回、容易验证，可直接自动执行；
- **黄区：** 会改变外部状态但仍可撤回，先生成方案并通过规则验证；
- **红区：** 高金额、合同、封号、公开发布等不可逆动作，只能准备材料，必须人工确认。

分级时要考虑可逆性、可验证性和责任归属，并把预算、次数、规则冲突、数据缺失和 Human Gate 写进系统。

## 八、最终公式

可靠的 Agent Loop = 外部状态 + 规划动作 + 工具执行 + 硬验证 + 可恢复记忆 + 停止条件。

自动化的终点，不是 AI 说“完成”，而是现实中的事情真的开始推进。真正的智能，不是替人逃避责任，而是把责任放在正确的位置。
